Subject: [isabelle] "Type variable has two distinct sorts" exception on locale interpretation

Dear list,

I have a strange exception when instantiating a locale, that seems to be
caused by accidentally overlapping type-variable names. The gist of it
is that interpretations with different type-name annotations cause
different (or no) exceptions.

In the original theory, I have also encountered cases where the
instantiation does not raise an exception, but yields a too special
type, by unrelated type variables with the same sort constraints being
unified.

A full example to reproduce the problem is below, here's the important
lines:

(*  Interpretation seems to depend on type-variable names
*)
interpretation foo: on_from_basic_indexable
  "basic_assn"
  "basic_extr"
  . (* Type variable has two distinct sorts, ?'a llvm_rep vs marker *)
  oops

interpretation foo: on_from_basic_indexable
  "basic_assn :: 'xxa::llvm_rep option list ⇒ 'xxa ptr ⇒ llvm_amemory ⇒
bool"
  "basic_extr"
  . (* Type variable has two distinct sorts, ?'a len2 vs marker *)
  oops

interpretation foo: on_from_basic_indexable
  "basic_assn :: 'xxa::llvm_rep option list ⇒ 'xxa ptr ⇒ llvm_amemory ⇒
bool"
  "basic_extr :: 'xxa ptr ⇒ 'xa::len2 word ⇒ 'xxa ptr llM"
  . (* works *)

In the example, I have used the marker type-class to pinpoint the
culprit inside the locale, that gets accidentally unified with some type
variables from the instantiation arguments (see below). It stems from a
definition statement inside the locale, that is subsequently used as
argument of a sublocale.

Any ideas what happens here? I'm currently trying to invent random type
names to work around that problem in my actual (more complicated) setup.

Best,

Peter
