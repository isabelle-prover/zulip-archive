Replacing the double-quotes properly is not so easy. For formally checked-text
it can be done reliably, based on authentic PIDE markup produced by
Isabelle/ML at the point where inner syntax is recognized. That will also work
for arbitrary nesting of embedded formal languages, e.g. in antiquotations.
That won't work for informal use of the same, because the markup is missing.

The traditional double-quoting of inner-syntax is not deprecated yet, just a
bit old-fashioned: I use it myself routinely, but not universally.

A few very elite users have switched their formal live to Isabelle cartouches
already some years ago. It looks better and is more robust. Disadvantage: It
is not immediately clear how to input it, despite the Isabelle/jEdit
completion on the " character. The German and French keyboards should also
have a standard mapping for it that I never know.

Dear Diego,
I was able to reproduce the problem in a small theory, and it looks like a bug or at least a missing registration in the interaction between setup_lifting and Finite_Map.
What happens is that setup_lifting turns your typedef into an internal Quotient theorem and then tries to generate a parametrized correspondence relation for the raw type fmap. For that, it consults the registered quot_map theorem for fmap. The existing one from Finite_Map is not in the shape that this part of Lifting expects for a type copy of fmap, which causes the low-level dest_state exception.
A working workaround is to locally override the quot_map theorem for fmap with a two-premise version, and additionally register the relator facts that setup_lifting needs:
theory Scratch
imports "HOL-Library.Finite_Map"
begin
typedef myfmap =
"UNIV :: ((nat, nat) fmap) set"
..
lemma Quotient_fmap_bnf_2prem[quot_map]:
assumes "Quotient (=) (id :: 'a => 'a) id (=)"
assumes "Quotient R Abs Rep T"
shows "Quotient (fmrel R)
(fmmap Abs :: ('a, 'b) fmap => ('a, 'c) fmap)
(fmmap Rep :: ('a, 'c) fmap => ('a, 'b) fmap)
(fmrel T)"
using assms Quotient_fmap_bnf
by (simp add: Quotient_alt_def)
declare fmap.rel_eq[relator_eq]
declare fmap.rel_mono[relator_mono]
declare fmap.rel_compp[symmetric, relator_distr]
lemma left_total_fmrel_eq[transfer_rule]: "left_total (fmrel (=))"
by (simp add: fmap.rel_eq left_total_eq)
lemma bi_total_fmrel_eq[transfer_rule]: "bi_total (fmrel (=))"
by (simp add: fmap.rel_eq bi_total_eq)
lemma left_unique_fmrel_eq[transfer_rule]: "left_unique (fmrel (=))"
by (simp add: fmap.rel_eq left_unique_eq)
lemma right_unique_fmrel_eq[transfer_rule]: "right_unique (fmrel (=))"
by (simp add: fmap.rel_eq right_unique_eq)
lemma bi_unique_fmrel_eq[transfer_rule]: "bi_unique (fmrel (=))"
by (simp add: fmap.rel_eq bi_unique_eq)
lemma right_total_fmrel_eq[transfer_rule]: "right_total (fmrel (=))"
by (simp add: fmap.rel_eq right_total_eq)
setup_lifting type_definition_myfmap
end
With these declarations in place, setup_lifting type_definition_myfmap succeeds cleanly.
So in short:

  • your original typedef is fine;
  • the failure is caused by how Lifting reuses the registered fmap quotient-map information;
  • the workaround above makes the registrations explicit enough for this case.
    This seems worth reporting upstream, since Finite_Map already has the relevant BNF facts, but not in a form that setup_lifting can directly use here.
    Best,
    Diego
